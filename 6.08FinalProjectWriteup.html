Overview:

For our final project in 6.08, we have designed and implemented three modules that serve to ‘smartify’ students’ dorm rooms. In addition to servicing the IoT and electronics side, we have also tried to expand our reach by incorporating dynamic mechanisms, a mobile app for user interfacing, and 3D printing enclosures in an attempt to replicate what a final product on the market could look like.

The first module we introduce in this write-up is the knock-detection and camera module. When a person knocks on a user’s door, a picture of them is taken, processed onto the server, and is sent to the app. We illustrate in the demo video one of many scenarios where this could be useful.

Next is the door lock module. The idea is that by bringing the door handle of the door down just enough with the servo, people on the other side can come through when the door is by default in a locked state. This is one of two modules with mechanical capabilities.

Our last module is the one responsible for toggling both the light switch in a room and a connected RGB LED strip. It switches a light switch on and off using a compliant mechanism which we 3D printed at the Cypress Engineering Design Studio. By interfacing with the server via the app, users are able to set specific RGB values or select a preprogrammed light sequence.

Last but not least, the app is vital in bringing all the modules together under one interface. Written in ReactNative, the app is meant to make the smart dorm modules an extension of a user’s pre-existing IoT environments by making it accessible on their phones.

Knock-detection/Camera Module:

Fig. 2: State Machine Diagram

Our first goal with the camera was to ensure reliable picture taking at the ESP level, not yet worrying about sending the image to the server. We needed to correctly set up the camera on the breadboard, and were able to correctly implement the starting code to get the camera to take pictures after pressing the capture button on an HTML web form.

The next step was to send the image that was taken to a server where the image could then be used for other purposes. In order to accomplish this, we decided that instead of using an HTML web form and small image buffers that send the image to the HTML form, it would be better to take away the web form functionality, and store the whole image to one buffer. Now that we had the image in one buffer, and the picture taking capabilities weren’t controlled by the webform, this opened up the ability to take pictures when some condition was met.

Once we had the full image buffer, we then tried to encode it and create a new array of the decoded string. However, there was not enough memory to store both arrays if the picture resolution was high, so we lowered the resolution and were able to safely store the base 64 character array. We then sent the encoded string to the server in JSON format. On the server side, we took the data that came from the POST request, decoded the string, and then wrote the binary data to a JPG file in order to get an image. This image lives on the server and is also displayed on the app.

For the knock-detection part of the module, we knew that either movement or sound based thresholds would not be reliable enough and could be triggered by ambient signal noise. As such, the knock detection is correlation based, where 13 recorded audio and IMU samples are compared to a stream of incoming data. As shown in the state diagram, we only pick from the stream of microphone and IMU data if the IMU detects an initial jolt, presumably from rattling the door.

At first, we were only running correlation on audio data with a threshold of 73% similarity. However, once we started comparing IMU data during the knock as well, our knock-detection became much more consistent without having to lower the correlation minimum.

An unexpected issue we encountered when integrating the knock-detection with the camera was that one ESP does not have enough on-board storage for buffer allocation. To solve this problem, we ended up having to use two ESP’s and when a knock was detected on one ESP, it would send a digitalWrite, alerting the other to a knock. We decided this simple approach to communication would be more robust than constant GET and POST requests via the server.

Door Lock Module:

Fig. 3: State Machine Diagram

The first design challenge we ran into was getting the correct PWM signal. We did a lot of research to find out that the correct duty cycle range for this servo is 5-10% with a frequency of 100 Hz. Even when we put this in we wouldn’t get the expected speed of 0 when we fed in the duty cycle of 7.5%. This just meant that we had to play around with the values a little bit until we got it right. After that it was a matter of making a method that transformed a value from 0 to 180 to an action that the motor did. Then we implemented the state machine to call that function.

The code has two methods write_to_servo() and run_door_handle(). write_to_servo() takes a value from 0 to 180 and moves the servo; 0 moves it full speed backwards, 90 stops it, and 180 moves it full speed forwards.

void write_to_servo(int angle){
 if(angle == 90){
   ledcWrite(1, 0);
 }
 else{
 int duty = 0; //have duty express the duty cycle as a 0% to 100% value for display.
 duty = map(angle, 0, 180, 6, 24);
 int val = 0;
 val = map(duty, 0,100, 0,4095);
 ledcWrite(1, val);
}

run_door_handle() runs the state machine for moving the door handle and checks whether the door should be unlocked or not.

void run_door_handle(){

 if(handle_state == IDLE){
   write_to_servo(90);
   Serial.println("IDLE");
   if(unlock){
     handle_state = OPENING;
     open_timer = millis();
   }
 }else if(handle_state == OPENING){
   Serial.println("OPENING");
   if(millis()-open_timer < open_duration){
     write_to_servo(170);
   }
   else{
     handle_state = HOLDING;
     hold_timer = millis();
   }
 }else if(handle_state == HOLDING){

   }
   else{
     handle_state = CLOSING;
     close_timer = millis();
   }
 }else if(handle_state == CLOSING){
   Serial.println("CLOSING");
   if(millis()-close_timer < close_duration){
     write_to_servo(50);
   }

Light Control Module:

Fig. 4: State Machine Diagram

The design challenges were primarily related to the hardware of the module. Achieving a sufficient power supply while also wiring up the ESP correctly served to be somewhat difficult. However, we were able to cut a USB cord in order to make a 5V power supply. In regards to the code for the light module, the process was relatively straightforward. There were some challenges regarding the frequency of requests to the server, as they were found to disrupt the different animations on the lights. However, to completely fix this issue we need some kind of support for parallelized code, so to fix this issue we simply made requests to the server less frequent. This way, animations would be affected to a minimum, while still being responsive to requests from the app.

Mobile App:

  • Condensing unused features - The app features three sliders to control the amount of red, green, and blue used in the LED lights and to offer the full spectrum of color. However, these sliders take up much real estate on the home screen, which can be overwhelming if the LED lights are not in use. As such, when the LED lights are turned off, the panel displaying the color sliders disappears. The app also features buttons that allow the user to change the mode of the LED lights (flashing, gradient, etc). These buttons also disappear when the LED lights are toggled off.
  • App background color - A cool, light blue is used as the background color of the app because blue conveys a state of calmness, relaxation, and comfort that is desirable for a Smart Home app.

React Native offers support for loading resources from a remote URL, so POST and GET requests were feasible additions to this project. Week 1 (Abe)

Ultrasonic Sensor Threshold Setup:

We ultimately decided not to use an ultrasonic sensor to be our threshold for the camera module as we later found that using the IMU to detect the door rattling would save us from adding another sensor to our set up. We figured simplicity was our best bet here, especially as our code became more complicated as we thought about future integration.

Week 4 (Ty)

Facial Recognition on Server Images:

Due to some complications with the library on the server, we were unable to get facial recognition working beyond local testing. As cool as it would have been, this was not part of our original plan; in this way, it points to a possible new feature in future iterations. Additionally, we would likely need to store images with a greater resolution to be able to reliably run facial recognition.

Reflection:

After having completed the project, we collectively feel much more comfortable with collaborative coding and hardware integration. Additionally, we found that the structured layout of weekly milestones was crucial in keeping us accountable for the project’s progress; ultimately what this meant was that we didn’t leave things until the last minute, at least in terms of technical work. We also learned that working as a team, especially at MIT, can be tough, with everyone’s busy schedules making it difficult to find times to work.

Regardless, we have left this project satisfied with the last four weeks worth of work. Given more time, we would have tried further pursuing the facial recognition route on the server, and perhaps adding another module for opening and closing blinds.

Overall, we feel that this project does a great job at showcasing our diverse skill set, especially with elements that lie outside of strictly electronics such as our mechanisms and mobile app. After a few weeks of work, we have created something that very well could be used by students in their dorm rooms, and seeing it all come together in the end was extremely exciting!
